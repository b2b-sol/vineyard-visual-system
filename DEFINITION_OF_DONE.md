# Definition of Done

The project is complete only when all sections below pass.

## Repository and operations

- Public repository exists at `b2b-sol/vineyard-visual-system`.
- All long-running GitHub activity is performed as `tiklebot`.
- Main branch uses passing required checks and merge queue or a documented equivalent if GitHub changed the feature.
- GitHub Pages hosts the atlas.
- The repository can also build locally from a clean checkout.
- Control files accurately report complete status.

## Workflow model

- All 10 priority workflow families are modeled.
- At least 10 complete swimlanes exist.
- At least 8 meaningful record/workflow state machines exist.
- At least 10 exception or decision trees exist.
- Operating model, annual calendar, roles map, handoff/information map, contract-fruit comparison, and certification overlay exist.
- Every workflow includes confidence and evidence classification.

## Scenario coverage

- At least 70 substantive scenario cards exist, with the nine scenario categories applied where relevant.
- Scenarios use realistic roles, records, timing, organizations, constraints, and completion conditions.
- Cross-organization, offline/delayed, correction, blocked, and audit scenarios are represented.

## Product structure

- Product domain map, scope boundary, information architecture, navigation models, role/surface matrix, mobile/desktop matrix, permission/approval model, notification/escalation model, screen registry, and state matrix exist.
- Every registered screen traces to scenarios and workflows.
- Every consequential action maps to a workflow state transition or decision.

## Synthetic operation

- Connected estate and contract-fruit data exists.
- At least one full season of representative records exists.
- At least 15 intentionally difficult exception histories exist.
- Referential integrity, chronology, units, and status vocabulary validate.

## Wireframes and high-fidelity design

- Every priority workflow has a complete low-fidelity flow.
- At least five end-to-end clickable prototypes cover the required prototype priorities.
- At least 50 distinct high-fidelity screen compositions exist across mobile and desktop, excluding trivial duplicates.
- At least 100 meaningful rendered state variants exist across the library.
- Consequential flows include non-ideal states.
- Review sheets and annotations exist.

## Design system

- Approved visual thesis and principles exist.
- DTCG token package exists and validates.
- At least 30 reusable components or coherent component families exist with documented variants and states.
- Icon, map-symbol, chart, typography, spacing, density, status, and content systems exist.
- Approved screens are composed primarily from the documented system rather than one-off styling.

## Construction library

- At least 15 coherent construction packets cover the five production waves.
- Every packet contains the inputs listed in `WORKFLOW_PRODUCTION_PLAN.md`.
- Screen and component specifications are machine-readable or consistently structured.

## Figma readiness

- Required `figma/` bundle exists.
- Tokens, icons, diagrams, screens, components, and prototypes have stable maps.
- Future MCP handoff prompt and migration checklist exist.
- Canonical designs remain structured and editable.

## Validation

- Type, lint, schema, unit, build, browser, responsive, accessibility, link, traceability, and repository-audit checks pass.
- No unresolved P0/P1 findings.
- Completion auditor signs off.
- Public atlas demonstrates the breadth above rather than hiding unimplemented requirements behind documentation.

## Final deliverables

- Public Pages URL
- Repository README with local setup and atlas tour
- `validation/COMPLETION_REPORT.md`
- final artifact inventory
- unresolved field-validation questions clearly separated from project incompleteness
- tagged release with generated release notes
