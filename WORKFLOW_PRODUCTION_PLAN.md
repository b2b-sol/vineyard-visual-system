# Workflow-Grounded Production Plan

> Execution revision, 2026-07-13: the evidence and ontology foundations remain ordered, but later work now runs as five complete vertical production waves. See `PACKET_REVIEW.md` and `control/EXECUTION_PLAN.md`. Those live documents supersede purely serial sequencing where they conflict with this original plan.

## Phase 0 — Build the factory

Create the repository structure, atlas application, schemas, validators, screenshot tooling, manifests, task system, custom agents, CI, and review surfaces. The factory is complete when a sample artifact can be registered, rendered, validated, screenshot, linked through traceability, and displayed in the atlas.

## Phase 1 — Structured source extraction

Run two independent extractors over the complete workflow report. Each returns workflows, roles, records, decisions, handoffs, exceptions, seasonality, compliance overlays, confidence, and source anchors. A Sol adjudicator reconciles disagreements.

Required registries:

- workflow registry
- role registry
- record registry
- decision registry
- exception registry
- terminology glossary
- evidence map

## Phase 2 — Workflow atlas

Create:

- operating-model diagram
- annual workflow calendar
- role interaction map
- field/office/outside-party information map
- estate-grown versus contract-fruit comparison
- certification overlay map
- swimlanes for all priority workflows
- record state machines
- decision and exception trees

Priority workflow families:

1. Seasonal work planning and dispatch
2. Scouting to treatment recommendation, approval, application, and recordkeeping
3. Irrigation scheduling and fertility adjustment
4. Crop estimation and forecast revision
5. Maturity sampling and winery communication
6. Pick decision and harvest planning
7. Harvest execution, delivery, weigh ticket, quality, and settlement reconciliation
8. Crew scheduling, task verification, attendance, payroll/cost allocation
9. Materials, purchasing, equipment readiness, and repair exceptions
10. Grower onboarding, contract-fruit coordination, certification, audit, and corrective action

## Phase 3 — Scenario catalog

For each applicable priority workflow, produce scenarios covering:

- routine
- urgent
- blocked
- incomplete information
- correction
- cross-organization
- offline/delayed communication
- supervisory review
- audit/historical review

Each scenario records trigger, role, context, goal, available/missing information, decisions, branches, records, communication, device, connectivity, compliance, and completion condition.

## Phase 4 — Product derivation

Create:

- product domain map
- scope and integration boundary
- role-to-surface matrix
- mobile/desktop responsibility matrix
- information architecture
- navigation models
- screen registry
- screen relationship graph
- permission and approval model
- notification/escalation model
- screen-state matrix

Every screen must trace to a scenario. Every major action must advance or resolve a workflow state.

## Phase 5 — Connected synthetic operation

Create a realistic multi-estate operation with estate-grown and contract fruit. Include intentionally messy but coherent histories:

- aliases and changed block identifiers
- planned/actual/verified work
- partial tasks
- cancelled application for wind
- REI/PHI restrictions
- offline observation synced later
- revised crop estimates
- maturity trends and late samples
- winery delivery-slot change
- wrong load identifier
- downgrade/rejection
- corrected weigh ticket
- incomplete audit record
- corrective action

Generate JSON/CSV fixtures, narrative context, copy deck, units, formats, statuses, and exception messages.

## Phase 6 — Low-fidelity workflow slices

For each workflow family create complete task flows, screen-flow strips, wireframes, state variants, and annotations. Use two alternative layout agents for high-value workflows, a critic, and an integrator.

## Phase 7 — Visual language

Produce three distinct visual directions. Each includes a design thesis, typography, color/status system, density, surfaces, map treatment, tables, charts, icons, mobile sample, desktop sample, and operational record. Score and select autonomously.

## Phase 8 — High-fidelity workflow packages

Build screen families by workflow rather than isolated pages. Include queues, details, create/edit, map, schedule, approval, execution, verification, exception, history, mobile counterpart, desktop counterpart, and print/export where relevant.

## Phase 9 — Design system extraction

Extract repeated approved patterns into components and tokens. Define purpose, data, variants, states, interactions, content constraints, responsive behavior, accessibility, examples, and invalid usage.

## Phase 10 — Construction packets

Create one packet per buildable workflow slice, including scope, workflows, scenarios, screens, approved visuals, component references, interaction sequence, states, permissions, sample data, copy, assets, acceptance scenarios, dependencies, exclusions, and definition of done.

## Phase 11 — End-to-end prototypes and validation

Prototype at least:

1. Plan → assign → execute → verify
2. Scout → recommend → approve → apply → record → follow up
3. Estimate → sample → decide pick → schedule harvest
4. Harvest → transport → weigh → assess → reconcile
5. Detect exception → escalate → decide → correct → preserve history

Run workflow-fidelity, UX, operational-data, accessibility/field-use, and implementation-feasibility reviewers.

## Phase 12 — Final integration

Repair cross-workflow inconsistencies, orphan traceability, duplicated terminology, visual drift, broken prototypes, missing states, and weak construction packets. Produce the final completion report and public release.
