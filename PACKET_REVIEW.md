# Autonomous Packet Review

Date: 2026-07-13  
Status: approved for execution  
Review method: parent synthesis plus independent product/workflow, infrastructure, and Definition-of-Done reviews

## What remains strong

- The product is correctly framed as a coordination, identity, and record-integrity system rather than a catalog of viticulture topics.
- Stable block identity, time context, restrictions, handoffs, correction history, and auditability remain foundational.
- The evidence hierarchy and confidence classifications remain mandatory.
- The vineyard-centered boundary with explicit winery, payroll, accounting, lab, contractor, and certification integrations is sound.
- Connected synthetic data, normal and non-ideal states, editable canonical sources, browser validation, creator-critic-integrator loops, and future Figma compatibility remain non-negotiable.
- The quantitative Definition of Done remains a coverage floor, subject to substantive-purpose and quality checks.

## What changes

- Replace the post-foundation waterfall with five vertical production waves. Each wave carries evidence, workflows, scenarios, fixtures, low/high fidelity, components, prototype behavior, construction packets, traceability, review, and rendered validation together.
- Move deterministic synthetic identity, status, unit, and event contracts before final scenario production.
- Split design-system work into early foundations and continuous extraction from accepted workflow slices.
- Replace the wide traceability CSV as canonical storage with normalized node and edge registries; generate human-readable views.
- Expand the task registry into an executable dependency graph with acceptance criteria, gates, artifacts, checks, timestamps, and Git references.
- Add a machine-readable DoD matrix, findings lifecycle, coverage matrices, generation-freshness checks, and state-consistency validation.
- Treat the original packet inventory as provenance and generate the live artifact inventory separately.
- Normalize public documentation to UTF-8 and replace opaque source handles with a durable evidence ledger as source normalization proceeds.

## Explicit project decisions

1. The atlas is an implementation-grade design and reference system with simulated interactions, not production farm-management software.
2. The synthetic operation uses a 2026 California North Coast composite, Pacific time, US customary operating units with SI equivalents where useful, and explicit non-legal-advice labeling for compliance examples.
3. The canonical taxonomy is ten workflow families plus cross-cutting overlays for identity, exceptions, permissions, notifications, offline/sync, and seasonal hazards.
4. The browser deliverable is a static React/TypeScript atlas using Pages-safe hash routing. Offline and conflict behavior are modeled and testable rather than backed by a production sync service.
5. Accessibility targets WCAG 2.2 AA plus outdoor readability, keyboard operation, reduced motion, touch targets, interruption recovery, and concise multilingual crew-facing copy where demonstrated.
6. Canonical artifacts remain structured. Selected review images are committed; exhaustive screenshots and reports may be retained as CI artifacts.
7. Figma completion means validated tokens, SVG assets, stable maps, component metadata, screenshots, and migration instructions; a live paid-plan Figma file is not required.
8. Numeric targets are minimum coverage floors and cannot be met with trivial duplicates, placeholder content, or untraceable variants.

## Missing mechanisms being added

- Schemas and semantic validators for evidence, roles, records, decisions, exceptions, requirements, transitions, findings, traces, prototypes, packets, Figma maps, and synthetic events.
- Seeded data generation with referential, chronology, status, vocabulary, and unit invariants.
- Route and coverage manifests for workflows, scenario categories, roles, platforms, states, components, packets, and prototypes.
- Stable required CI checks for quality, browser validation, and repository audit on `pull_request`, `merge_group`, and `main`.
- Screenshot provenance/freshness, Pages deep-link smoke tests, print checks, performance budgets, and manual field-use/accessibility rubrics.
- A structured P0-P3 finding, waiver, resolution, and recheck lifecycle.

## Revised route to completion

1. Repair control state and record this review.
2. Import the reviewed packet as the first bot-owned commit.
3. Build a deterministic factory and one complete walking slice from evidence to rendered screen and construction packet.
4. Establish CI, Pages, and repository governance after the first check names report.
5. Normalize evidence; run independent workflow extraction and adjudication; freeze the shared ontology and state vocabulary.
6. Generate the connected synthetic-season backbone, then finalize scenarios and product architecture against real fixture IDs.
7. Produce and score three visual directions on lighthouse workflow screens; autonomously select the strongest thesis.
8. Build five vertical waves: planning/crew; scouting/protection/irrigation; estimate/maturity; harvest/reconciliation; materials/growers/certification, with cross-cutting overlays in every wave.
9. Run a midpoint integration review, then consolidate the design system, construction library, Figma bundle, and coverage matrices.
10. Run machine, workflow, UX, visual, accessibility, data, implementation, and independent completion audits; repair all blocking findings; publish Pages and a tagged release.
